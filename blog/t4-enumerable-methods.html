Efficiency with Enumerables:

Enumerables & the Cycle Method

Technical Post #4

Enumerables appeared as a topic last week as some unknown Class/Module that we were supposed to use in one exercise while refactoring our code. At the time, I was using Enumerable methods without really knowing what an enumerable was, or why I was able to use its methods on other classes. There is no .map method in Arrays, but somehow I could use this method on them. I'm pretty use I'd get an error message if I tried using a string method on an integer, or vice versa.

It turns out, who reason Enumerables & its methods are a bit more flexible is because it is related to Modules. As a short explanation, Modules basically solve the problem of multiple inheritance (common in other languages) for Ruby. Classes only allow for single inheritance, and Modules act like containers for groups of methods from different classes, allowing them to get 'mixed-in' together. Another interesting fact about Enumerables I discovered is that Classes can only use Enumerables if it as a .each method defined. This is important to note when we think about which of our basic classes (or classes we create) we may want to use Enumerables with. If .each doesn't exist naturally, we would have to first define this method.

So what exactly is 'Enumerables'? Enumerables is pretty much just a class full of loops & iterators! As the Ruby Bastards Book explains, this MODULE provides a set of methods to traverse, search, sort and manipulate collections. Enumerable methods provide easier, more efficient and more capable ways of traversing through code.

Out of the Enumerable methods available to us, there are a couple that I find to be particularly useful so far.

  • each_with_index- functions just like our favorite .each method, but also allows for reference to the index
  • cycle - this method acts like a .times with a .each inside of it. Cycle takes an array, pulls out each element and executes it in the code block, then continues through every element in the array. THEN, when it's done, it goes back and does it again! Cycle takes an optional parameter (along with the code block), that tells it how many times to cycle through the code. If you don't give it a number, it will cycle through infinitely. You can break this infinite cycle with an IF clause for when to BREAK. Keep in mind, that because cycle acts like .EACH inside each loop, it does not alter the actual array element and return an altered array.
  • map - this is useful when you want to manipulate each element in an array, and have the new array returned without changing the original. This is different from .each because each doesn't actually alter the element(s) in the iteration. With .each, you would have to use .each_index (assuming you were staying away from enumerable methods) to manipulate and then reassign to that index. Map also has the nifty addition with ! on the end of the method, which alters the original array. Let's see this method in action:
    • zodiac = ['rat','ox','tiger','rabbit','dragon','snake','horse','sheep','monkey','rooster','dog','pig'].map do |zodiac_animal|
      • animal.upcase
    • end
    • puts zodiac.join(", ")

Now that you've gotten to see how .map works, go back and try out .cycle and .each_with_index! Are there any other methods that you favor in the Enumerable module?
