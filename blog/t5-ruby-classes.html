Ruby Classes

Scope and Organization in Ruby

It's in human nature to organize and classify. We have taken the world around us and provided words to refer to them, and then bigger words to refer to groups of things, and even bigger words to refer to groups of groups of things. This propensity to judge, label and categorize is something we do immediately, instinctively, whether we want to or not.

Considering how immediate and natural categorizing things is to humans, it only makes sense that when creating a new language, humans would create some way of grouping and categorizing things! In Ruby, they call it CLASSES (well, Modules also group things, but that's a topic for another day).

Think about what it would be like if we weren't allowed to group and classify things, and to talk about these groups and make assumptions about the group (what they are like, what they can do, etc) as a whole. Let's take PEOPLE for instance. We refer to people as a group, or class, and hold some basic assumptions about them. Special cases aside, we assume they all use their mouths to speak and eat, their eyes to see, their feet to walk, their hands to grab and hold. These are just a few of the basic assumptions that come to mind when you see something and classify it in your mind as HUMAN or PERSON.

Let's use HUMAN as our Ruby class, as an example. We'd define our class like this:

  • class Human
  • end
Now, there are classes, and then there are subclasses. For instance, HUMAN is a subclass of MAMMALS, which is a subclass of ANIMAL, which is a subclass of LIVING_THINGS. You could clarify this relationship as follows:
  • class Human < Mammals
    • end

You can also initialize an instance of a method. A specific person is an instance of a method. You'd create the new person with (here I'm giving him a name and an age):
  • John_Adams = Human.new("John Adams, 50")

Within the class, you'd have to include some code so that the class would know what it starts out with. Think of this as the things you are born with. When you are born, you have a name, an age (it's zero at that point, but still), parents. Depending on how you think of people, you might consider 'occupation' a defining trait throughout someone's life. These are your initializing variables.

Attr_accessor allows you to set and then return a variable. The initialize method allows you to initialize certain variables with certain values (either set or given as parameters).

  • class Human
    • attr_accessor :age, :name
    • def initialize
    • @age = age
    • @name = name
    • end
You see that @ symbol in front of the variable names? That means that it's an INSTANCE VARIABLE, not a local variable. Local variables only exist within a method. Instance variables can exist between all methods in the same class, but only for one instance of that class. Let's go back to the example to clarify. @age is an instance variable. I can use it in later methods under the Human Class without having to bring it in as a parameter next to the method name. However, John will not have the same as as Jill or Jack, so in that sense, while all 3 have been initialized with the instance variable @age, the instance variable only applies to one instance (ie: John), not ALL humans.

For things that apply universally to ALL humans (or to whatever class as a whole), there is a third kind of variable, and a third kind of method to go along with it: GLOBAL METHODS/VARIABLES. These methods and variables apply to all instances/people, and are acted out on the class as a whole (the human race). You can't have instance variables inside a global method, because global methods don't have anything to do with specific instances. An initialize method is actually an example of a global method: it's not specific to Jim, John or James. You .new a class, not a class object.

An instance method, on the other hand, refers to those methods that are acted out on the specific instance/class object. For instance, let's say we want to give Mr. Adams a job. We'd need to create an instance method inside the human class to set (or change) occupations. We could also use this to print information about the person, change things, age him, etc.

  • def occupation=(job)
  • @job = job
  • end
  • def occupation(job)
  • @job
  • end

So to summarize, we've discussed:

  • what is a class, why we need them and how to create one
  • subclasses
  • initializing a class
  • instance variables
  • attr_accessor
  • instance methods
  • global methods & variables
